# cppm — ClearPass NAC

Aruba ClearPass Policy Manager spoke. Repo: `cppm`. `module_type = "nac"`. See [architecture-topology.md](architecture-topology.md).

## Role & module_type

NAC spoke — endpoint auditing, session/access-tracker monitoring, and a hub-orchestrated IPAM→ClearPass endpoint sync. Source for the realtime NAC→IPAM reverse sync (`CPPM_GET_RECENT_SESSIONS` → `NETBOX_SYNC_ACCESS_TRACKER` on the netbox spoke).

## What it does

cppm is the Network Access Control (NAC) integration for Aruba ClearPass Policy Manager: it audits known endpoints, watches live authentication sessions (the Access Tracker), and keeps ClearPass and NetBox in sync in both directions — pushing IPAM-known endpoints into ClearPass ahead of auth, and pulling recently-authenticated sessions back into NetBox so the inventory reflects what's actually on the network.

In the WebUI this shows up under the **Security/NAC** area — a "NAC (Aruba ClearPass Policy Manager)" module tile, plus tabs for NAC Status, Access Tracker, My Devices, and Unknown Devices; Setup has a "NAC / ClearPass Instances" tile for connecting a ClearPass server, and System → Sync has the IPAM↔NAC sync schedules.

## Entrypoints

`python3 -m src.control_plane` (`CPPMControlPlane`); spoke `CPPMSpoke` (**no base class** — not `BaseSpoke`); falls back to `run_standalone_mode()` (FastAPI `0.0.0.0:8000`, `/status`) if `--hub` omitted. systemd `lm-cppm.service`. Installer `install.sh` (clones to `/opt/lm/cppm`, venv, `.env`, unit; `PYTHONPATH=$INSTALL_DIR:$INSTALL_DIR/core/src:$INSTALL_DIR/cppm/src`). `src/main.py` is a standalone demo script, not the entrypoint.

> **Primarily a role now.** cppm runs mainly as the **`cppm`** role hosted by the generic agent (`agent-<hostname>`, unit `lm-agent`): the agent opens a sub-spoke `{agent}-cppm` (module_type `nac`, parent-auto-approved) and self-installs it via `agent/src/agent_spoke.py::_install_role` (clones `lbockenstedt/cppm.git` + deps); because `CPPMSpoke` is non-BaseSpoke the role loader wraps it with `_RoleAdapter`. The dedicated `lm-cppm.service` / `install.sh` `cppm-spoke-1` path is the **legacy/standalone** alternative. Connection config (`CPPM_HOST`/creds) comes from the hub push (WebUI), not a per-module `.env`.

## Ports / backends

Talks to ClearPass REST (`CPPMClient`, `src/client.py`, `requests.Session`, TLS verify off). Auth: OAuth2 (`POST /api/oauth`) with **password grant preferred over client_credentials** when user creds are available (inherits the user's operator profile vs the API client's restricted profile); falls back to basic auth. Token cached with expiry (`expires_in`, 30s skew). Endpoints: `/api/session` (access tracker / recent sessions / user sessions / NAC status), `/api/endpoint` (device DB, by-MAC, by-IP, upsert/sync), `/api/role`. Standalone mode serves :8000.

`GET_VERSION`, `CPPM_REFRESH_CACHE`, `UPDATE_CONFIG`, `TEST_AUTH` (tries every OAuth candidate, reports per-attempt), `PROBE_API` (raw `_request`). Cached: `CPPM_GET_ACCESS_TRACKER`, `CPPM_GET_DEVICE_DATABASE`, `CPPM_GET_NAC_STATUS`. Live: `CPPM_GET_RECENT_SESSIONS` (last `lookback_minutes` default 2 — **not cached**, time-sensitive, called ~60s by the hub realtime NAC→IPAM loop), `CPPM_GET_SYSTEM_HEALTH`, `GET_DEVICE` (by MAC), `LIST_ENDPOINTS`, `GET_ENDPOINT_DETAIL`, `GET_DEVICE_SESSIONS`, `GET_USER_SESSIONS`, `GET_LOGS` (auth logs by start/end), `LIST_ROLES`, `SEARCH_SESSIONS`, `CPPM_SYNC_ENDPOINTS` (hub-orchestrated IPAM→ClearPass: upserts a tenant's endpoint batch tagged `NetBox_Tenant_Slug`/`_Name`/`_ID` + `Tenant`/`Tenant_Slug` + `IP Address`/`Hostname`/`status:Known`; `replace=True` deletes endpoints previously tagged with this tenant absent from the batch; MAC-keyed upsert with IP fallback; IP-only records with no existing endpoint skipped; per-endpoint failures counted, never raised).

## Notable behaviors & gotchas

- **`CPPMSpoke` is non-BaseSpoke** — `class CPPMSpoke:` (no `BaseSpoke` base). Implements `spoke_id`/`config`/`get_version`/`handle_command` itself; registered into the control plane's `modules` dict. The control plane also locally overrides `register_module` (same pattern as ldap). On the hub side, the agent-spoke role loader wraps it with `_RoleAdapter`.
- **Endpoint IP fallback scan** — ClearPass's `ip_address` filter only matches the first-class field; `_endpoint_ips` collects IPs from arbitrary attribute names and `_build_ip/mac_endpoint_map` do bounded paged scans (`ENDPOINT_SCAN_CAP` / a larger sync cap) so static-IP devices whose IP lives under a profiler-populated attribute are still matched and their MAC reused.
- **In-memory cache** for `CPPM_GET_ACCESS_TRACKER`/`CPPM_GET_DEVICE_DATABASE`/`CPPM_GET_NAC_STATUS` only; `CPPM_GET_RECENT_SESSIONS` deliberately uncached (realtime loop).
- **Token strategy** — password grant with `client_id` (configured, then `ClearPass`), then password grant with no `client_id`, then `client_credentials` (user operator profile > API client profile). Tolerates empty 204 bodies on DELETE.
- **`CPPM_SYNC_ENDPOINTS` tenant tagging** uses the same attribute names the at-auth-time Context Server Action uses (`NetBox_Tenant_Slug` etc.) so an Enforcement Policy matches the tenant whether the endpoint was synced in advance or tagged at auth time.

## How it works

- **Connection config comes from the hub push, not a `.env`.** `UPDATE_CONFIG` carries `host`/`user`/`password`/`client_id`/`client_secret`; the spoke calls `CPPMClient.update_config()`, which clears any cached OAuth token so the next request re-authenticates against the new config. `CPPM_HOST`/`CPPM_CLIENT_ID`/`CPPM_CLIENT_SECRET`/`CPPM_USER`/`CPPM_PASS` env vars / `.env.example` are only a bootstrap fallback for standalone/dev use.
- **OAuth token strategy.** ClearPass's `/api/oauth` is tried with a preference order, not a single fixed grant: if user credentials are configured, **password grant is tried first** — once with the configured `client_id` (if any), then again with `client_id="ClearPass"`, then with no `client_id` at all — because password grant inherits the authenticating user's own operator profile rather than the API client's (often more restricted) profile. `client_credentials` (client_id + client_secret) is the fallback if password grant isn't configured or fails. A successful token is cached with its `expires_in` and refreshed ~30s before expiry. `TEST_AUTH` runs through every candidate and reports the outcome of each attempt individually, rather than just the first success/failure — this is the fastest way to see exactly which grant type ClearPass is accepting or rejecting.
- **Cached vs. live reads.** `CPPM_GET_ACCESS_TRACKER`, `CPPM_GET_DEVICE_DATABASE`, and `CPPM_GET_NAC_STATUS` are served from an in-memory cache when the request uses default paging/filtering (no `limit`/`offset`/`status` in the request) and a cache entry exists; `CPPM_REFRESH_CACHE` (or the WebUI's own refresh action) re-populates all three from ClearPass. Any paged or filtered request (page 2, a specific `status`) always goes live — the cache only serves the default view, so filtering never hands back a stale default page.
- **`CPPM_GET_RECENT_SESSIONS` is deliberately never cached** — it's a realtime pull of sessions started in the last `lookback_minutes` (default 2), and it's what the hub's background **realtime NAC→IPAM reverse sync** loop calls roughly every 60 seconds (`lm/core/src/realtime_ipam_nac_sync.py`, config key `realtime_ipam_nac_sync`, loop starts ~75s after hub startup). Each cycle: pull recent sessions from the connected NAC spoke → attribute each session's IP to a tenant by NetBox prefix containment → push the attributed sessions to the netbox spoke via `NETBOX_SYNC_ACCESS_TRACKER` with `replace=False` (only-add-missing — NetBox stays authoritative, this sync never overwrites or deletes a hand-managed NetBox record). Sessions with no MAC are dropped before the push (nothing to match/create by); sessions whose IP falls in no tenant's prefix are dropped and counted as unattributed.
- **The forward direction — IPAM → ClearPass — is `CPPM_SYNC_ENDPOINTS`,** driven by the hub's `endpoint_sync.py` on its own schedule (config key `netbox_cppm_sync`; interval or daily-time mode). Each cycle the hub pulls a tenant's IPs from NetBox (or another registered IPAM source) and pushes the batch to the cppm spoke, which upserts each endpoint into ClearPass Device Inventory tagged with `NetBox_Tenant_Slug`/`NetBox_Tenant_Name`/`NetBox_Tenant_ID` plus friendly `Tenant`/`Tenant_Slug` attributes — the same attribute names the at-auth-time Context Server Action stamps, so an Enforcement Policy matches the tenant whether the endpoint was pre-synced or tagged live at auth. Matching is MAC-first (a batch-built MAC→endpoint map avoids one HTTP round-trip per record); IP-only records are resolved by an existing ClearPass endpoint carrying that IP under any attribute name (not just the first-class `ip_address` field ClearPass's REST filter is limited to), or, failing that, a MAC borrowed from a live session whose `framedipaddress` matches. Records that still have no MAC after both attempts are skipped, since ClearPass endpoints are MAC-keyed. `replace=True` (the default) makes IPAM authoritative: endpoints previously tagged for this tenant but absent from the current batch are deleted.
- **Sync loops summary:** endpoint sync (IPAM→ClearPass, `CPPM_SYNC_ENDPOINTS`) runs on its own configurable schedule from `endpoint_sync.py`; the realtime reverse sync (ClearPass→NetBox, `CPPM_GET_RECENT_SESSIONS` → `NETBOX_SYNC_ACCESS_TRACKER`) runs roughly every 60 seconds from `realtime_ipam_nac_sync.py`. Both are hub-scheduled — the cppm spoke itself has no timers of its own, it only answers each request when asked.
- **Every ClearPass call runs in a thread executor** (`run_in_executor`) because `CPPMQueries`/`CPPMClient` use synchronous `requests`, not async I/O — this keeps a slow/hung ClearPass call from blocking the spoke's event loop.

## How to use it

- **Connect ClearPass.** Setup → NAC / ClearPass Instances → add/edit the instance: host (hostname or IP; `https://` is assumed if omitted), plus either an API client id/secret, or a username/password (or both — password grant is preferred automatically when both are present). Saving pushes `UPDATE_CONFIG` to the connected cppm spoke immediately.
- **Verify the connection is actually working.** Use TEST_AUTH (exposed as a "Test" / "Test Auth" action against the instance) — it tries every OAuth candidate ClearPass would accept given the configured credentials and reports each attempt's HTTP status/response individually, so you can see exactly which grant type is failing rather than a single opaque error.
- **Force a fresh pull of endpoints/sessions/status** instead of waiting on the cache: use the refresh action tied to `CPPM_REFRESH_CACHE`, or open a paged/filtered view (which always bypasses the cache).
- **Trigger an IPAM→ClearPass endpoint sync on demand** for one tenant or all tenants from System → Sync's endpoint-sync card — this runs `CPPM_SYNC_ENDPOINTS` immediately instead of waiting for the scheduled interval/daily time.
- **Trigger the realtime NAC→IPAM reverse sync on demand** ("Sync now" on the realtime NAC→IPAM card in the same System → Sync area) to pull the last `lookback_minutes` of sessions and push newly-seen devices into NetBox right away.
- **Look up a specific device or session**: use My Devices / Unknown Devices / Access Tracker tabs, which map to `GET_DEVICE`/`GET_ENDPOINT_DETAIL` (by MAC), `GET_DEVICE_SESSIONS`/`GET_USER_SESSIONS`, and `SEARCH_SESSIONS`/global search respectively.

## Troubleshooting / common questions

- **"Auth to ClearPass keeps failing — how do I tell why?"** Run TEST_AUTH. It reports every OAuth attempt tried (password grant with each candidate `client_id`, then `client_credentials`) with that attempt's actual HTTP result — an `HTTP 400`/`401` on the password-grant attempts usually means the ClearPass Guest/API Login username or password is wrong or the operator profile lacks API access; a failure only on the `client_credentials` attempt usually means the API client's client_id/secret is wrong or the client's operator profile is too restricted. Compare against what's actually configured on the ClearPass side (API Services → API Clients, and the Operator Login the username belongs to).
- **"The NAC spoke shows offline/red in Diagnostics — what breaks?"** Both sync loops (endpoint sync and the realtime reverse sync) skip a cycle cleanly when no NAC spoke is connected — they don't error loudly, they just log and wait for the next cycle. Cached reads (Access Tracker/Device Database/NAC Status) keep serving whatever was last cached; anything requiring a live call (recent sessions, a paged/filtered query, TEST_AUTH) will fail until the spoke reconnects.
- **"An endpoint isn't getting tagged with the right tenant / isn't showing up in ClearPass at all."** If the record only has an IP (no MAC) on the NetBox side, `CPPM_SYNC_ENDPOINTS` can only tag it if ClearPass already has an endpoint carrying that IP (under any attribute, not just the standard `ip_address` field) or a live session whose `framedipaddress` matches — check the sync's per-tenant result for a skipped count and reason. If it has neither, ClearPass has no MAC to key the endpoint on and the record is skipped; the fix is to make sure the device authenticates at least once (creating a ClearPass session/endpoint) before it can be synced by IP alone.
- **"A tenant's endpoints disappeared from ClearPass after a sync."** `CPPM_SYNC_ENDPOINTS` runs with `replace=True` by default — IPAM is treated as authoritative, so any endpoint previously tagged for that tenant but no longer present in the current NetBox batch is deleted from ClearPass. Check whether the device was removed/moved off that tenant's prefix in NetBox first.
- **"Recent sessions aren't showing up in NetBox."** The realtime reverse sync only adds — it never overwrites or deletes NetBox records, and it drops any session whose IP doesn't fall inside one of the target tenant's NetBox prefixes (check for an "unattributed" count) or that has no MAC at all. Also confirm both a NAC (cppm) spoke and an IPAM (netbox) spoke are connected — the loop needs both to run a cycle.
- **"Is cppm running as its own service or as part of the agent?"** In the current topology, cppm normally runs as the `cppm` role inside the generic `lm-agent` unit (sub-spoke `{agent}-cppm`, auto-approved by its parent, wrapped in a `_RoleAdapter` because `CPPMSpoke` isn't a `BaseSpoke`) — check Diagnostics for a spoke id like `agent-<hostname>-cppm`. The standalone `lm-cppm.service` / `install.sh` path (spoke id `cppm-spoke-1`) still works but is the legacy path; either way, connection config always arrives via the hub push, never a local `.env`.
