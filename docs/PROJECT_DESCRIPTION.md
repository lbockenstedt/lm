# Lab Manager - Project Description

## Overview
Lab Manager is a modular, containerized, **multitenant** application designed to provide a **single pane of glass** interface for managing and monitoring various infrastructure components. It is essentially a "Private Cloud Orchestrator" (similar to AWS or Azure) for local lab resources; users specify their resource requirements (intent), and the system handles the backend implementation across the hardware pool.

## Architecture
- **Hub-and-Spoke Design**: A central hub acting as an aggregation point for various spoke modules. In this architecture, "Module" is synonymous with "Spoke".
- **Messaging Protocol (Mailbox Method)**:
  - Communication occurs via **WebSockets on port 443** using a "mailbox" pattern.
  - **Push-Ack-Retry**: Every message must be acknowledged. No fire-and-forget.
  - **Bidirectional Flow**: The Hub acts as the central switchboard. Spokes connect and poll/receive commands from their mailbox, while pushing updates to the Hub's inbound mailbox.
  - **Reliability**: The Hub queues messages for offline spokes for up to 24 hours.
  - **Health Monitoring (Traffic Light)**: Heartbeats every 30 seconds.
    - Green: Healthy.
    - Yellow (Warning): No heartbeat $\ge 120$s.
    - Red (Offline): No heartbeat $\ge 300$s.
- **Modular Design & Deployment Flexibility**: Each spoke is independent and can be deployed via LXC, Docker, or OS installation.
- **API-First Approach**: Every operation must expose an API; a WebUI module will be provided for user interaction.
## Security & Key Management
- **Mutual Authentication**: Required between the Hub and every Spoke/Module.
  - **Centralized Key Management**: All API keys are managed by the Hub.
- **Encryption at Rest**:
  - Sensitive data is fully encrypted at rest using a certificate generated by the Hub.
  - Admin can download/upload this key to rotate it (Decrypt $\rightarrow$ Re-encrypt).
- **Key Rotation**: Keys rotate every 7 days; previous 4 keys remain valid for recovery/backup purposes.
- **Onboarding**: New spokes use a "First Secret" generated in the Hub to automatically onboard and authenticate.
## Resource Governance & Approval Policies
- **Standard Builds (Auto-Approved)**: Pre-defined resource profiles can be provisioned automatically.
- **Custom/High-Resource Builds (Approval Required)**: Requests exceeding thresholds require explicit Admin approval before orchestration begins.
- **Concurrency**: Global lock on tasks; one task processed at a time. Users are queued and notified upon completion.

## Orchestration & Workflow
- **Workflow Logic**: Hard-coded, well-defined sequences for infrastructure provisioning.
- **Atomic Transactions**: Any failure in a multi-step chain triggers a full rollback of all preceding steps to prevent "zombie" resources.
- **Auditing & Reporting**: Detailed error reports must include:
  - What failed and exactly what changes were rolled back.
  - A "Before vs. After" configuration comparison for diagnostic purposes.
## Multitenancy & UI Customization
- **Multi-Tenant Support**: Shared pool of infrastructure services.
- **Tenant Mapping**: Isolation via Proxmox labels and NetBox Tenant IDs.
- **Dynamic Quotas**: Resource quotas are dynamic and configurable per tenant.
- **UI Skinning**: Per-tenant visual themes (CSS, Logos, Colors).
- **Dynamic Dashboards**: Widgets are added/removed based on the modules active for that specific tenant.
## Authentication & Access Control
- **Identity Management**: Modular integration. LDAP is the initial target but can be replaced by AD or Azure Auth via custom spokes.
- **User Administration**: Dedicated frontend interface to manage the users and group permissions.
- **Emergency Access**: Local static accounts are available per tenant as a "break-glass" mechanism if the auth spoke fails.

## Data Management & State
- **Source of Truth (SoT)**: The Hub is NOT the SoT. Authoritative data remains in NetBox, Proxmox, OPNSense, etc.
- **Memory-Centric**: Primary state is held in memory for performance.
- **Persistence**: Memory state is backed by a JSON cache on disk for cold starts.
- **Cache TTL**: Data is refreshed from spokes every $\le 2$ minutes.

## QA & Documentation Standards
- **Definition of Done (DoD)**: A feature is complete only when it has:
  1. Functional Logic implemented.
  2. Inline code documentation/docstrings.
  3. Updated User Manual/README in `docs/`.
  4. Passing test cases in the QA module.
- **QA Auditor**: The QA module is a deployable agent (LXC/Docker) that acts as a "Ghost Tenant," testing the entire API and deployment chain from an external perspective.
